1.  Explica brevemente el flujo de trabajo **nuevo** completo que implementaste con **Terraform** (commit -> CI -> Build/Push Imagen -> Deploy TF Staging -> Update Service Staging -> Test Staging -> Deploy TF Prod -> Update Service Prod -> Smoke Test Prod). Sé *específico* sobre *qué artefacto se mueve, qué hace cada job principal, y qué valida cada tipo de prueba*.
- Commit: Se realiza un commit en el repositorio de código, lo que desencadena el pipeline de CI/CD.
- CI: El pipeline de CI/CD se activa, ejecutando los jobs definidos en el archivo de configuración del pipeline.
- Build/Push Imagen: Se construye una nueva imagen Docker de la aplicación y se sube a Docker Hub.
- Deploy TF Staging: Se despliega la infraestructura en el entorno de Staging utilizando Terraform, creando los recursos necesarios para ejecutar la aplicación.
- Update Service Staging: Se actualiza el servicio en Staging para que apunte a la nueva imagen Docker desplegada.
- Test Staging: Se ejecutan pruebas automatizadas en el entorno de Staging para validar que la aplicación funciona correctamente antes de desplegar a Producción.
- Deploy TF Prod: Si las pruebas en Staging son exitosas, se despliega la
infraestructura en el entorno de Producción utilizando Terraform.
- Update Service Prod: Se actualiza el servicio en Producción para que apunte a la
nueva imagen Docker desplegada.
- Smoke Test Prod: Se ejecutan pruebas de humo en el entorno de Producción para validar que la aplicación está funcionando correctamente después del despliegue.
2.  ¿Qué ventajas y desventajas encontraste al usar Terraform o infraestructura como código en vez de desplegar manualmente? ¿Qué te pareció definir la infraestructura en HCL?
- Ventajas: Permite automatizar y versionar la infraestructura, lo que mejora la reproducibilidad y facilita el mantenimiento. Además, Terraform es compatible con múltiples proveedores de nube, lo que brinda flexibilidad. Definir la infraestructura en HCL es intuitivo y fácil de leer, lo que facilita la colaboración entre equipos.
- Desventajas: Requiere una curva de aprendizaje inicial para entender cómo funciona Terraform y HCL. Además, los errores en la configuración pueden llevar a despliegues fallidos o a la creación de recursos no deseados, lo que puede generar costos adicionales. También, la depuración de errores en Terraform puede ser más compleja que en despliegues manuales, especialmente para aquellos que no están familiarizados con la herramienta.
3.  ¿Qué ventajas y desventajas tiene introducir un entorno de Staging en el pipeline de despliegue a AWS? ¿Cómo impacta esto la velocidad vs. la seguridad del despliegue?
- Ventajas: Introducir un entorno de Staging permite validar los cambios en un entorno que simula Producción antes de desplegar, lo que reduce el riesgo de errores críticos en Producción. Además, permite realizar pruebas más exhaustivas y detectar problemas de integración o rendimiento antes de afectar a los usuarios finales.
- Desventajas: Agregar un entorno de Staging puede aumentar el tiempo total del pipeline, ya que se deben ejecutar pruebas adicionales y realizar despliegues en dos entornos diferentes. Esto puede relentizar la velocidad de despliegue, especialmente si las pruebas en Staging son extensas o si el proceso de despliegue es complejo.
4.  ¿Qué diferencia hay entre las pruebas ejecutadas contra Staging (`test-staging`) y las ejecutadas contra Producción (`smoke-test-production`) en tu pipeline? ¿Por qué esta diferencia?
- Las pruebas ejecutadas contra Staging (`test-staging`) son pruebas más exhaustivas que pueden incluir pruebas unitarias, de integración y de rendimiento para validar que la aplicación funciona correctamente en un entorno similar a Producción. Por otro lado, las pruebas ejecutadas contra Producción (`smoke-test-production`) son pruebas de humo, que son pruebas rápidas y básicas para verificar que la aplicación está funcionando después del despliegue. La diferencia radica en el nivel de detalle y el alcance de las pruebas, ya que en Staging se busca validar la funcionalidad completa, mientras que en Producción se busca validar la estabilidad básica después del despliegue.

- Sería mejor montar un funcionalidad que nos permita, ejecutar solo parte del pipeline para cambios pequeños, por ejemplo, solo ejecutar el despliegue a staging y las pruebas en staging, sin necesidad de desplegar a producción cada vez que se hace un cambio menor. Esto podría implementarse utilizando etiquetas o condiciones en el pipeline para determinar qué partes del pipeline se deben ejecutar según el tipo de cambio realizado. Sin embargo, es importante asegurarse de que los cambios críticos aún pasen por todo el pipeline para garantizar la calidad y estabilidad de la aplicación en Producción.
